# <a name="define-your-mobile-engagement-strategy"></a>Mobile Engagement 戦略の定義
*アプリを作成する理由は、ユーザーに使用してもらうことです。*

ユーザーが好む優れたアプリを作成しようと多大な努力をしていることと思います。 また、ユーザーを獲得するためにおそらくかなりのマーケティング予算を投資していることでしょう。 しかし、当初のユーザーの刺激がピークに達した後、アプリが徐々に使用されなくなることがあります。 "*Azure Mobile Engagement の意義はまさにそこにあります*"。ユーザーをとどまらせ、テストと学習を通じてアプリを段階的に改善できるのです。

リテンション期間と使用率の向上に対する私たちの取り組みは、プッシュ通知とアプリ内メッセージを通じてアプリ ユーザーの関心を引くことに基づいていますが、これはアプリ内でのユーザーの行動に合わせて調整したメッセージと通信を使用して非常に特殊な方法で行われています。 私たちの目標は、適切なタイミングと適切な場所で適切な対象ユーザーとの通信を可能にすることです。

ただし、そのためには、最初に *ユーザーを理解*してから、ユーザーが行ったことまたはユーザーの特性 (セグメントと呼びます) に基づいてグループを作成し、さらに各セグメントに関係のある通信を作成する必要があります。

## <a name="mobile-engagement-serves-your-objectives"></a>Mobile Engagement による目標の達成
*保有期間と使用率について説明しましたが、その目的は何でしょうか。*

Mobile Engagement 戦略を構築するには、最初にアプリの目標と主要業績評価指標 (KPI) を確認する必要があります。

最初にこれらの目標と KPI を定義すると、適切なエンゲージメント ユース ケースを定義できます。

ユース ケースとは、単純なようこそから、IT システムによってトリガーされる非常に高度なユーティリティ通知まで、ユーザーに知らせたいキャンペーンの単純なリストです。 適切に構築されたユース ケースには、少なくとも *いつ、何を、だれに*という 3 つの項目が含まれます。

1. 非常に短い説明 ("ようこそキャンペーン" など)。
2. **何を**: メッセージの例 (たとえば、"ご利用いただきありがとうございます。 1 か月間無料でご利用いただくには、ログインしてください" など)。 このメッセージは最終的なものではなく、いつでも変更できますが、通常は何を知らせたいかについて考え始めるのに役立ちます。
3. **だれに**: このメッセージを受信するセグメント (たとえば、"3 日前に初めてアプリを起動し、ログイン ページにアクセスしたが、まだログインしていないすべてのユーザー")。
   * これは Azure Mobile Engagement で非常に簡単に行うことができます。
   * ここでも、セグメントはいつでも定義できるため、これを最終的なものにする必要はありませんが、セグメント化の条件を早い段階に定義して、適切なデータを収集することが重要です。
4. **いつ**: キャンペーンのタイミングです。 これは、特定の日、特定の操作の後、またはトリガーに基づくことができます。 Mobile Engagement では、通信を適切なタイミングで行うことができる可能性がかなり高くなります。

ユース ケースとセグメントを定義すると、アプリケーション内で収集する必要があるデータを定義するためのガイドラインが提供されます。 これは、 *"タグ計画"*のロールです。 タグ計画では、データ収集が開発者に対して指定されていることを確認できます。 そのため開発者は、キャンペーンを適切なデータで操作するのに適したセットアップで Mobile Engagement を埋め込みます。 統合が正しく、必要なものが収集されることを確認するテストを実行することも非常に重要です。

統合に基づいて、アプリケーションが発行されると、マーケティング担当者はリアルタイムで分析を参照し、対象ユーザーをセグメント化してから、スマート ターゲット プッシュ通知の送信を開始してアプリの内外でエンド ユーザーと情報交換できるようになります。

### <a name="use-cases-to-get-started"></a>開始のユース ケース
1. ようこそ戦略: 最初のセッションから 2/5/10/15 日後に再び関心を引き、最初の実行のリテンション期間を延ばすために、アプリケーションの起動時にエンド ユーザーの行動に基づいていくつかのプッシュ通知キャンペーンを作成します。
2. エンド ユーザーの行動に基づいて新しいコンテンツ (機能、記事/ビデオ、製品) をプロモートして、関心を引く可能性の高い情報だけをエンド ユーザーに送信します。
3. アプリケーションの評価: ストアでアプリを 5 つ星として評価する可能性が最も高い、ユーザー ベースの 1% 未満を対象とします。
4. サブスクリプションの促進: 貴重なコンテンツをまだ見たことのないエンドユーザーにプロモートして、サブスクリプションを増やします。
5. チュートリアル: すべてのユーザーに必須のチュートリアルは不要です。 アプリ内に優れたチュートリアルを構築し、ユーザーがアプリを使用していないと思われる場合や機能の使用に問題がある場合にのみアプリ内メッセージでそれらを作動させましょう。

## <a name="why-do-you-need-analytics-to-engage"></a>関心を引くために分析が必要な理由
既に理解されているかもしれませんが、プッシュ通知のブロードキャストだけでは不十分です。 Mobile Engagement のコアとなる概念は、マーケティング担当者と開発者が適切なタイミングと場所で適切なエンド ユーザーにかかわるようにすることです。 この 3 つの主要概念を理解するには、アプリケーションから分析を収集し、対象ユーザーのセグメント化に使用することが不可欠です。 これは、行動セグメントが他のデータベースまたは CRM、あるいはクロス チャネルからのデータを補完する場合にさらに強力になります。 Mobile Engagement では、任意の場所からデータを取得し、それを使用して適切なユーザーにターゲットを絞ることができます。

対象ユーザーの関心を引く際にできるだけ状況に即したものにするには、エンド ユーザーの行動を理解して、その状況をリアルタイムで把握することが非常に重要です。 データ収集により、マーケティング担当者はユース ケースにとって重要な項目に重点を置き、モバイル エンゲージメント戦略目標を達成できます。 これまでに設定した目標の達成は、実際のベスト プラクティスが分析であらゆるデータを収集することではなく、調べたい項目とユース ケースに重点を置くことができるデータのみを収集することである理由にもなります。 これは、ソリューションの使用方法を開始、試行、テスト、学習し、スマート プッシュ通知に対処して、アプリケーションのリテンション期間を延ばして成功に導くための優れた方法です。

> [!NOTE]
> データが多すぎるとデータが生かされないことを忘れないでください。

### <a name="use-cases-and-best-practices"></a>ユース ケースとベスト プラクティス
以降のセクションでは、作業の開始に役立てるために、弊社のお客様から報告された主要なユース ケースについて簡単に説明します。

#### <a name="media"></a>メディア
エンド ユーザーによって使用されるコンテンツの種類を収集し、この行動に基づいて対象ユーザーをセグメント化して、特定のタイプのコンテンツのターゲットを、それを使用する可能性の高い対象ユーザーのみに設定します。 ユーザー ベース全体へのスパムは回避し、リテンション期間を向上させます。

#### <a name="m-commerce"></a>M コマース
アプリケーションおよび対象ユーザー内で最もアクセスの多かった製品カテゴリを収集し、エンド ユーザーが購入する可能性の高い、そのカテゴリ内の割引または新製品をプロモートします。 収益の向上を目指します。 この場合も、目標はスパムをしないことです。

#### <a name="gaming"></a>ゲーム
エンド ユーザーのゲーム レベルと、特定の期間に費やされた時間を収集して、行き詰まっている可能性がある対象ユーザーや、特典の提供により次のレベルに進む可能性の高い対象ユーザーをターゲットにします。

しばらくの間プレイしていないユーザーに特定のイベントをインセンティブと共に伝え、再びプレイしてもらうことを試みます。

#### <a name="retail"></a>小売
対象ユーザーが消費する可能性の高い製品やブランドをお気に入りや行動に基づいて収集し、対象ユーザーを店舗に導き、売上の増加を図ります。

#### <a name="banking"></a>銀行取引
アプリケーションの初回起動時に口座を作成したエンド ユーザーからデータを収集します。 ターゲットを絞ったプッシュ通知でようこそ戦略をデプロイし、口座のサブスクリプション数を増やすことを目指します。

### <a name="how-to-create-a-great-tag-plan"></a>優れたタグ計画の作成方法
タグ計画は、ユーザーパスの説明またはアプリケーションのワークフローの一種のようなものにして、ユーザーの行動を理解し、ユーザー ベースを適切にセグメント化するのに十分な分析を行うために収集する必要のあるすべてのタグ (データ) を提供する必要があります。 これは技術的なプロセスではありません。 そのため、マーケティング担当者は、Mobile Engagement 戦略に基づいて収集するデータを指定できます。

少なくとも、アプリケーションのすべての画面 (Mobile Engagement では *アクティビティ* と呼ばれる) にタグを付けます。 これにより、ユーザー パスを特定できます。

アクティビティには、ボタンのクリックなどの操作情報を収集する *イベント* を埋め込むことができます。 これにより、アプリケーション内での操作を収集できます。 したがって、マーケティング担当者は、ユーザーがアクセスしている画面や実行している操作を把握できます。

`Jobs` は期間のあるアクションです。 これは、ユーザーがアカウントの作成やログインなどを実行するまでにかかった時間をマーケット担当者が把握するのに非常に役立ちます。 また、開発者が Web サービスの呼び出しまでにかかった時間を監視するのにも役立ちます。

`Errors` を監視して、ユーザーがアプリで問題を抱えているかどうかを知ることもできます。 たとえば、接続の問題が頻繁に発生する場合などです。

このようなデータはすべて、アプリケーションから動的データを収集できるパラメーター (Mobile Engagement では*追加情報* ) で補強できます。 これは、きめ細かいセグメント化を可能にするために重要です。 たとえば、マーケティング担当者は、ユーザーが使用するコンテンツの種類に基づいてユーザーをセグメント化できます。 コンテンツの種類は、アクティビティまたはイベントの動的な情報です。

*アプリ情報* は、アプリケーションまたはユーザーの状態をリアルタイムで把握できるようにするデータです。 これは、対象のユーザー ベースを分類して迅速にターゲットを設定する際にも役立ちます。 たとえば、ユーザーがログインしているかどうかという状態を示す true/false やサブスクリプションの有効期限などです。

#### <a name="example-of-tags"></a>タグの例
*ユースケース: 対象ユーザーの行動をセグメント化して、適切な内容のプッシュ通知を適切なエンド ユーザーに送信します。*

1. プッシュ通知を送信し、製品のカテゴリをプロモートする: 行動データを収集して、特定の期間に x 回アクセスした製品のカテゴリやカートに追加した特定のアイテムに基づいて対象ユーザーをセグメント化します。 収集したデータにより、セグメント化を行い、適切な対象ユーザーにプッシュ通知を送信できるようになります。
2. アプリを評価する: ソーシャル ネットワーク上の対象ユーザーが共有しているコンテンツに基づいてデータを収集します。 アプリの " *大使* " を特定することで対象ユーザーをセグメント化することを目指します。 アプリ内のプッシュ通知を使用する場合、アプリの大使は、ストアでアプリに 5 つ星評価を付けるように依頼するのに最適なアプリの対象ユーザーです。
   
   ![][1]

*ユースケース: 宣言型データ*

1. アラート ニュースのセグメント化: 宣言型データを収集し、その設定に基づいて対象ユーザーをセグメント化します。 これにより、特定の対象ユーザーが本当に興味のある特定のトピックのプッシュ通知を送信できます。
2. ログイン ステータスに基づいて対象ユーザーをセグメント化します。 データを収集して、ユーザーが接続しているかどうか、またはアカウントを作成したかどうかを確認します。 まだログインしていないエンド ユーザーにターゲットを絞り、エンド ユーザーの気持ちを変えるプッシュ通知を送信するのに役立ちます。
